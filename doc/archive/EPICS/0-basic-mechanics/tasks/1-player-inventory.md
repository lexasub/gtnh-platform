# TASK: Инвентарь игрока (Player Inventory)

**Layer**: 1
**Статус**: Draft
**Эпик**: 0-basic-mechanics

## Affected Services

| Service | Role | R/W |
|---------|------|-----|
| **MetaDB** (Go/SQLite) | Primary — хранит и загружает инвентарь игрока (blob) | Read/Write |
| **Gateway** | Relay — загружает инвентарь при логине, шлёт InventoryUpdate клиенту | Relay |
| **SimulationCore** | Post-hook — подписывается на события инвентаря для геймплея | Event subscriber |
| **GameClient** | Consumer — отображает инвентарь (ImGui), отправляет действия со слотами | Read/Write |

> **Архитектурное правило**: Инвентарём владеет MetaDB. SimulationCore НЕ управляет состоянием инвентаря — только подписывается на события. Flow: Client ↔ Gateway ↔ MetaDB (CRUD), и MetaDB публикует события → SimulationCore реагирует.

---

## Описание

Реализация инвентаря игрока: хотбар 10 слотов, полный инвентарь 4×10, открытие/закрытие клавишами E/Escape.

---

## Требования

### Хотбар
- Полоска из 10 слотов внизу экрана
- Выбор активного слота: клавиши 1–9 (0 для десятого)
- Выбранный слот визуально выделен

### Полный инвентарь
- Клавиша **E** открывает/закрывает окно инвентаря
- Окно: 4 строки × 10 слотов
- Слоты хотбара отображаются внизу окна инвентаря
- Выход: Escape или повторное E

### Ограничения
- **Нет** области крафта в интерфейсе игрока (крафт — только через верстак)
- Реализация должна быть подменяема (чтобы можно было увеличить размер)
- **Не рисовать** руки/тело игрока (только UI инвентаря)

### Персистентность
- Инвентарь хранится в MetaDB как сериализованный blob (или отдельная таблица)
- `user_id = 0` (MVP — один пользователь)
- **Login**: Gateway → GetInventory → MetaDB → InventoryUpdate → Client
- **Runtime**: Client → PlayerAction → SimulationCore → ... → InventoryUpdate → Client
- **Logout**: Client → InventorySave → MetaDB → Commit()

### Операции со слотами
| Операция | Описание |
|----------|----------|
| Add | Добавить предмет в слот (с учётом стака) |
| Remove | Убрать предметы из слота |
| Transfer | Переместить между инвентарями |
| Swap | Обменять два слота |

## FlatBuffers Schema

```flatbuffers
table ItemStack {
    item_id: uint16;   // ID предмета (0 = пусто)
    count: uint8;      // количество (1–64)
    meta: uint16;      // прочность/состояние
}

table PlayerAction {
    player_id: uint64;
    action: PlayerActionType;  // PLACE, BREAK, MOVE, USE
    x: uint32; y: uint32; z: uint32;
    block_id: uint16;
    selected_slot: uint8;      // выбранный слот хотбара
}

table InventoryUpdate {
    player_id: uint64;
    slots: [ItemStack];        // полный снимок инвентаря
}
```

## Acceptance Criteria

#### Сценарий: Открыть и закрыть инвентарь
1. Игрок нажимает **E**
2. Открывается окно с 4×10 слотами + хотбар
3. Слоты пусты (или с начальным набором)
4. Игрок нажимает **Escape** — окно закрывается
5. Игрок снова нажимает **E** — окно снова открывается

#### Сценарий: Выбор слота хотбара
1. Игрок нажимает **3**
2. Третий слот хотбара визуально выделяется
3. Активный слот меняется на 3

#### Сценарий: Загрузка инвентаря при логине
1. Игрок подключается к Gateway
2. Gateway шлёт `GetInventory(0)` в MetaDB
3. MetaDB возвращает сериализованный blob
4. Gateway шлёт `InventoryUpdate` клиенту
5. Клиент отображает 45 слотов с предметами

#### Сценарий: Сохранение при логауте
1. Игрок отключается
2. Client шлёт `InventorySave`
3. MetaDB сериализует все слоты и пишет на диск
4. Gateway закрывает соединение
