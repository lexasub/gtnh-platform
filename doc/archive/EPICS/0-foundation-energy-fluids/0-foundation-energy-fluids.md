# Энергия и жидкости

**Эпик**: Foundation — Energy & Fluids  
**Слой**: L1 (базовый) + L2 (отложенный)  
**Статус**: L1 — в разработке, L2 — отложено

## Affected Services

| Service | Layer | Role |
|---------|-------|------|
| **EntityStateStore** ⬅️ **NEW** | L0 | Persistence — snapshots EnergyStorage + FluidTank state |
| **Gateway** | L0 | Relay — forwards energy data in EntitySnapshot |
| **SimulationCore** | L1 | Owns EnergyStorage ECS component, tracks `currentEnergy` per machine |
| **PipeNetwork** | L1 | Energy/liquid flow graph — resolves distribution each tick |
| **GameClient** | L1 | Consumer — displays energy/fluid levels in machine GUI |

> **Architecture rule**: PipeNetwork is independently accessible via MessageRouter. SimulationCore sends it block-change events and reads flow results, but others can query it directly.

## Обзор

Энергия и жидкости — два фундаментальных ресурса, которые связывают машины в функциональные сети. Энергия (EU — Energy Unit) питает механизмы, а жидкости переносят сырьё и продукты между блоками.

В **L1** реализуются примитивы: хранение EU в машинах, базовые жидкостные слоты.  
В **L2** (отложено) — полноценные электрические сети с проводами, генераторами и потерями, а также трубопроводная система с давлением и потоком.

## Энергия (Energy Unit)

**Что**: GregTech EU — целочисленное значение энергии.

**Где**:
- Компонент `EnergyStorage` в ECS машин
- Отдельный граф `PipeNetwork` для трубопроводов

**Как вписать**:
- Добавить поле `energy` в `EntitySnapshot` и сообщения `PipeNetwork` (задел уже есть в протоколе)
- В MVP хранить энергию в машине просто как число

### EnergyStorage Component

`EnergyStorage` — ECS-компонент, описывающий способность блока хранить, потреблять и отдавать энергию:

```cpp
struct EnergyStorage {
    int32_t capacity;      // Максимальное хранилище EU
    int32_t currentEnergy; // Текущие EU
    int32_t maxInput;      // Максимум EU за тик (ввод)
    int32_t maxOutput;     // Максимум EU за тик (вывод)
    int32_t tier;          // Уровень (0 = 1 EU/т, 1 = 10 EU/т и т.д.)
};
```

**Правила**:
- Потребление ограничено `maxInput` — машина не может за один тик взять больше EU, чем указано
- Отдача ограничена `maxOutput` — излишки не сбрасываются, а остаются в резерве
- На следующий тик `currentEnergy` не может превышать `capacity`
- Передача энергии между машинами — через `EnergyEmitter` / `EnergyReceiver` (L2)

## Электрические сети (Layer 2 — отложено)

**Что**: Полноценные EU-сети по образцу GregTech: генератор → провод → машина. Провода имеют потери, силу тока и напряжение.

### Архитектура

Электрические сети — расширение `PipeNetwork`, который пока заглушка. Реализация:

- **Графовая модель**: вершины — блоки-источники/потребители, рёбра — провода
- **Расчёт**: на каждом тике решать потокораспределение в сети
- **SimulationCore** строит граф, обновляет при изменениях блоков, передаёт `PipeNetwork` для расчёта

### Примитивы

- `EnergyEmitter` / `EnergyReceiver` компоненты в ECS
- Провод — блок с `meta`-данными о типе (ток, сопротивление)
- Генераторы (L2): преобразуют топливо/пар в EU
- Трансформаторы (L2): изменение напряжения

## Жидкости (Fluid Stack)

**Что**: Аналог `ItemStack`, но для жидкостей: `fluid_id` + `amount_mb` (миллилитров).

**Где**: машины, трубы. В будущем — отдельные резервуары.

**Как вписать**:
- Добавить `FluidStack` в протокол
- В инвентарях машин — слоты для жидкостей
- В MVP упростить: машина имеет входной и выходной танк

### FluidStack

```cpp
struct FluidStack {
    uint32_t fluidId;      // ID жидкости (0 = пусто)
    int32_t amount;        // Количество в миллилитрах (max 1000 = 1 мБ)
    int8_t  color;         // Цвет (опционально, для идентификации)
};
```

**Правила**:
- `amount = 0` → пустой слот
- Максимум 1000 мл (1 мБ) на слот
- `FluidStack` — неизменяемая структура (immutable), при модификации создавать новую

## Трубы и транспортировка жидкостей (Layer 2 — отложено)

**Что**: Трубы для пара, воды, химикатов и других жидкостей.

### Архитектура

Аналогично электричеству: отдельный граф жидкостей, не пересекающийся с энергографом.

**Компоненты**:
- `FluidHandler` — способность блока обрабатывать жидкости (ввод/вывод/хранение)
- `FluidPipe` — компонент трубы с внутренним объёмом и давлением

**Дополнительно**:
- Жидкости могут течь под давлением
- Разные типы жидкостей — разные `fluidId`
- Соединения: трубы соединяются автоматически при касании (L2)
