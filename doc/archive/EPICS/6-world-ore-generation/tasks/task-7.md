# Task 7: ChunkStore Persistence for Ores

## Objective
Ensure ore blocks generated by WorldGenerator are persisted in LMDB via ChunkStore's existing chunk save/load mechanism. Ores should survive restarts.

## Requirements

### 7.1 Verify existing chunk persistence works for ores
**Location**: `src/services/chunk_store/`

The existing `Chunk` structure stores `blocks[32³]` as `uint16_t`:
```cpp
// Chunk.h — from explore agent
struct Chunk {
    uint16_t blocks[32][32][32];  // block_id at each position
    uint8_t meta[32][32][32];     // metadata
    uint32_t multiblock[32][32][32];  // multiblock ID
};
```

Since ore blocks are just `uint16_t` values (e.g., 3=iron_ore, 5=gold_ore), they should persist automatically through the existing LMDB save/load pipeline.

**Verify**:
1. `ServerWorld::saveChunk()` serializes the full blocks array
2. `ServerWorld::loadChunk()` deserializes it
3. Ore block_id values survive save/load roundtrip

If there's any filtering or validation that rejects unknown block_ids, add ore IDs to the whitelist.

### 7.2 Integration test
The flow should be:
1. WorldGenerator generates chunk with ore blocks
2. Chunk saved to LMDB via existing save mechanism
3. Chunk loaded from LMDB — ore blocks present
4. Second load from LMDB (not regeneration) — same ore positions

### 7.3 Verify chunk regeneration
If a chunk fails to load from LMDB (corrupted or missing), it should be regenerated:
```cpp
// ServerWorld::GetChunk()
if (!loaded) {
    regenerate = true;
    // Generate terrain + ores via WorldGenerator
}
```

Verify that regeneration produces the SAME ore block positions (deterministic from seed).

## Acceptance Criteria
- [ ] Ore blocks survive chunk save/load in LMDB
- [ ] Ore values not filtered out by any validation
- [ ] Regenerated chunks produce deterministic ore positions
- [ ] Ore persistence does not increase chunk size beyond existing limits (32 KB blocks + 32 KB meta + 128 KB multiblock = 192 KB)
- [ ] No change to existing chunk format needed

## Dependencies
- Task 5 (WorldGenerator integration — ores placed in chunk data)
- ChunkStore (existing persistence layer)
